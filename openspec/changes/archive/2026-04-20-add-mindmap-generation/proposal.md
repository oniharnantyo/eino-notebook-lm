## Why

Users need a way to visually understand the structure and key concepts of ingested sources (PDFs, websites, text). Currently, the only way to explore source content is by reading the raw text or chatting with it via RAG. A mindmap provides an immediate structural overview, helping users quickly grasp the document's organization and main topics.

## What Changes

- Add a new **artifacts** system — a generic table and domain concept that stores LLM-generated outputs (mindmaps, future: podcasts, slides, etc.)
- Add an async mindmap generation endpoint: user triggers generation, backend processes via Gemini LLM, stores the result
- Add list/get artifact endpoints so users can retrieve generated artifacts
- The mindmap is generated by sending `Source.Content` to Gemini with a structured prompt, returning a hierarchical tree JSON

## Capabilities

### New Capabilities
- `artifact-management`: Generic artifact storage and lifecycle — create, list, get artifacts with type/status tracking. Covers the domain entity, repository, and CRUD API.
- `mindmap-generation`: Mindmap generation from source content — async LLM call, structured prompt, tree-shaped JSON output. Covers the usecase and trigger endpoint.

### Modified Capabilities
_(none — this is a new feature with no changes to existing spec-level behavior)_

## Impact

- **New domain entity**: `Artifact` in `internal/core/domain/entities/`
- **New repository interface**: `ArtifactRepository` in `internal/core/domain/repositories/`
- **New use cases**: `ArtifactUseCase` (CRUD) and `MindmapUseCase` (generation) in `internal/core/application/usecases/`
- **New HTTP handlers + routes**: Under `internal/interfaces/http/`
- **New database table**: `artifacts` (id, source_id, notebook_id, type, status, result JSONB, error, timestamps)
- **Dependencies**: Reuses existing Gemini model provider — no new external dependencies
- **API surface**: 3 new endpoints under `/api/v1/notebooks/{notebookId}/`
